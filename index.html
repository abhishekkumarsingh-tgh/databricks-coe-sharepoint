🧱 Lakehouse Architecture Explained
🥉🥈🥇
Medallion Architecture
Bronze (Raw): Append-only ingestion from all sources. Preserves raw data exactly as received. No transformations.

Silver (Cleansed): Data conformed, deduplicated, validated. Schema enforced. Business rules applied. One record of truth per entity.

Gold (Business): Aggregated, enriched, domain-specific datasets optimised for analytics and ML. Rebuilt from Silver on a scheduled cadence.
Architecture Guide →Standard
🔄
Batch vs Streaming
Batch: Scheduled Databricks Workflows or DLT pipelines. Best for large volumes with latency tolerance >5 minutes. Uses Autoloader for efficient incremental file ingestion.

Streaming: Structured Streaming or DLT with Event Hub / Kafka as source. Best for near-real-time requirements (seconds to minutes). Supports stateful operations, watermarking, and exactly-once semantics via Delta.

Decision guide and flow chart available in the reference architecture library →
Decision Guide →Architecture
🌊
Delta Lake ACID Principles
Atomicity: Writes either complete fully or do not appear at all. No partial writes.
Consistency: Schema enforcement prevents incompatible records. Schema evolution must be explicit.
Isolation: Concurrent reads/writes handled via optimistic concurrency and Delta log checkpoints.
Durability: Committed data is guaranteed to survive failures. Transaction log is durable.

Additional features: Time Travel, VACUUM, OPTIMIZE, Z-ORDER, Change Data Feed, Liquid Clustering.
Delta Lake Guide →Core Concept
⚡
Photon Engine Overview
Photon is a native vectorised query engine written in C++ that dramatically accelerates Spark SQL and DataFrame operations on Databricks Runtime 9.1+.

Best for: SQL queries, aggregations, joins, and window functions on large datasets.
Not suitable for: Python UDFs, Pandas UDFs, and custom serialisers (these revert to JVM).
Typical gains: 2–8× query speed improvement, up to 50% DBU reduction on SQL-heavy workloads.

Enable via cluster policy or runtime configuration. Required for BI SQL Warehouses.

📊 Workspace Types & Topology
🏗️
Dev / Test / Prod Setup
Standard pattern: 1 workspace per environment per business domain.

• Dev: Shared among team. No production data. Sandbox cluster policy. No auto-approval for library installs.
• Staging: Production mirror for validation. Minimal users — CI/CD service principal only + approvers.
• Prod: No interactive cluster access. Job clusters only. All changes via CI/CD pipeline.

Unity Catalog metastore is shared across all three workspaces in the same region.
Setup Guide →Standard
🌍
Multi-Region Strategy
Primary region: UK South (all prod workloads).
DR region: UK West (replication via Delta Sharing + ADF copy pipelines).

One Unity Catalog metastore per region — UK South metastore is the master. Cross-region catalog access is read-only using Delta Sharing.

Data residency: all PII and Restricted data must remain in UK South. Cross-subscription access requires Security approval.
DR Runbook →Reference
🕸️
Hub-Spoke Networking Model
All Databricks workspaces are deployed into spoke VNets using VNet Injection. Spokes are peered to a central hub VNet containing: shared DNS, firewall (Azure Firewall Premium), and express route / VPN gateway to on-prem.

No-public-IP is mandatory for all clusters. Private Link enabled for control plane and data plane. NSG rules permit only required Databricks service tags plus corporate CIDR.

Private Endpoints deployed for: ADLS Gen2, Azure Key Vault, Event Hub, and SQL Server sources.

💬 Objection Handling
❓
"Databricks is expensive"
Response: Compare total cost, not licence cost. Databricks customers typically reduce their overall data platform spend by 20–40% by eliminating: separate data warehouse licences, standalone ML platforms, ETL tool licences, and the engineering overhead of stitching them together. Use the CoE TCO Calculator — input their current stack cost and let the numbers respond.
❓
"We already use Snowflake"
Response: Databricks and Snowflake complement rather than replace each other in some architectures, but if they're building ML or AI capabilities they will hit Snowflake's limits (no native MLflow, no GPU clusters, no Feature Store). For organisations where data + ML need to converge, Databricks is the obvious choice. Offer a side-by-side demo showing a use case Snowflake cannot handle natively.
❓
"We prefer Azure Synapse"
Response: Synapse is a solid Microsoft-integrated option for SQL-heavy BI workloads but has significant gaps in ML, streaming, and at-scale data engineering. Microsoft's own internal teams use Databricks for advanced analytics. Show performance benchmarks (Photon vs Synapse SQL), governance gaps (no Unity Catalog equivalent), and the engineering productivity comparison (Databricks notebooks vs Synapse notebooks are not equivalent in capability).
❓
"Security concerns" / "Vendor lock-in"
Security Response: Databricks holds SOC 2 Type II, ISO 27001, and is HIPAA-eligible. Data always stays in your Azure subscription — Databricks control plane never touches your data. All networking is Private Link. Reference our Security & Governance section and offer a Security Review session.

Lock-in Response: Delta Lake and MLflow are open-source and vendor-agnostic. Your data in Parquet + Delta format can be read by Spark, Trino, DuckDB, and many other engines. You are not locked to Databricks — you're locked to open formats.

⚙️ Core Optimisation Policies
⏰
Cluster Auto-Termination Policy
Mandatory auto-termination for all interactive (all-purpose) clusters. Enforced via cluster policy — cannot be disabled by users. Thresholds by team tier: Sandbox: 20 min, Data Engineering dev: 30 min, BI/Analytics: 10 min, ML dev: 60 min. Monitoring script alerts on clusters active >4 hours and raises a P3 ticket automatically.
🔀
Job Cluster vs All-Purpose Cluster
Use Job Clusters for: All scheduled pipeline runs, CI/CD test runs, ML training jobs, DLT pipelines. Job clusters terminate after the run — no idle cost.

Use All-Purpose Clusters for: Development and exploration only. Never run scheduled jobs on all-purpose clusters. All-purpose clusters cost 2× the DBU rate of job clusters for the same compute.
⚡
Photon Usage Guidelines
Always enable Photon for: SQL Warehouses (mandatory), BI/reporting clusters, any workload running SQL aggregations, joins, or window functions on large tables.

Photon is NOT beneficial for: Python UDF-heavy workloads (UDFs revert to JVM), pandas API on Spark, ML training (GPU clusters use different runtime). Photon costs slightly more per DBU but reduces overall DBU consumption — always beneficial for SQL-heavy workloads.
💸
Spot Instance Strategy
Use spot (preemptible) for: All job clusters running batch ingestion and transformation pipelines. Target: 80% spot on job clusters. Implement checkpoint-based restart logic so spot evictions don't cause data loss (DLT handles this automatically).

Never use spot for: Driver nodes (use on-demand), streaming pipelines where spot eviction causes SLA breach, SQL Warehouses, any cluster supporting external ML model serving.
🔍 Idle Workspace Detection
Idle Detection Script — Weekly Scheduled Job
# Queries Databricks system tables to find idle resources
# Run weekly via Databricks Workflow — results to Delta table

SELECT workspace_id, cluster_id, cluster_name,
  last_activity_time,
  datediff(now(), last_activity_time) AS idle_days,
  monthly_dbu_cost
FROM system.billing.usage u
JOIN cluster_metadata c USING (cluster_id)
WHERE idle_days > 7
AND cluster_type = 'all_purpose'
ORDER BY monthly_dbu_cost DESC
Output: email alert to cluster owner + team lead for any cluster idle >7 days. Auto-terminate after 14 days idle with 48hr warning.

❓ Frequently Asked Questions
Q: When should I use DLT vs standard Databricks Workflows?
Use DLT when you need: declarative pipeline definitions, built-in data quality expectations, automatic retries and error handling, CDC APPLY CHANGES INTO, or streaming + batch in one pipeline definition. Use Workflows for: complex multi-task orchestration with conditional branching, jobs that are not data transformation pipelines (e.g. model training, reporting), and tasks that mix Python scripts, SQL, and DLT runs.
Q: How do I pass secrets to a DLT pipeline?
Use the DLT pipeline configuration block in your bundle YAML to reference Databricks Secrets: configuration: { "source_key": "{{secrets/my_scope/my_key}}" }. The pipeline reads the secret at runtime. Never hard-code secrets in DLT pipeline Python code.
Q: What's the approved way to run SQL schema migrations in CI/CD?
Use Databricks Asset Bundles with a dedicated SQL migration task in the Workflow definition. Alternatively, use a migration script in /src/migrations/ that runs ALTER TABLE, CREATE TABLE IF NOT EXISTS, and COMMENT ON TABLE statements via the Databricks SQL connector in the CI pipeline's integration test stage.
Q: Can I use Terraform to manage Unity Catalog grants?
Yes — use the Databricks Terraform provider resources: databricks_grants for table/schema/catalog grants, databricks_group for AAD group sync, and databricks_user for service principals. Store grant definitions in the /infra/unity_catalog/ directory. Grants managed by Terraform are the source of truth — avoid granting via UC UI in production environments.
